# r1 WASM walk ‚Äî comprehensive history

> Snapshot of the actual WASM hash timeline for every Soroban
> source we ingest. Built from r1's full Galexie archive walk
> (`s3://galexie-archive`, ledgers 50,457,424 ‚Üí 62,296,694)
> captured 2026-04-30, supplemented by Soroban-RPC current-state
> queries for never-upgraded contracts the walker missed.
>
> **Coverage: 540 / 543 contracts attributed across 8 sources
> (99.4 %)**, including all 11 Blend contracts. All 52 unique
> WASMs fetched + SHA-256-verified + disassembled + classified
> by exported function signatures.

## What this answers

Per protocol/oracle that produces canonical trades:

1. **Which contracts are deployed on mainnet** ‚Äî every factory,
   router, per-instance pool, and oracle contract.
2. **When each was deployed** ‚Äî first-seen ledger from the walk,
   or "pre-walk-window" for contracts on never-upgraded WASMs.
3. **When each was upgraded** ‚Äî every
   `update_current_contract_wasm` event captured as a contiguous
   ledger range per active WASM hash.
4. **What each upgrade resulted in** ‚Äî factory upgrades cascade
   onto pool instances deployed afterwards; existing instances
   stay on their old WASM until individually upgraded.
5. **What each WASM does** ‚Äî function exports + symbol strings
   from disassembly, classified into roles
   (`<source>-{factory,router,pool,multihop,oracle,...}`).

## Tooling gaps surfaced

1. **`stellaratlas-ops extract-wasm-from-galexie` returns MISSING for every hash.**
   Tested 2026-05-01 against multiple subsets of the 50 hashes and a 12M-ledger
   archive range. Tool reports zero matches. Workaround used in this audit: bypass
   the tool entirely, fetch WASM bytes via Soroban-RPC `getLedgerEntries` against
   the ContractCode LedgerKey. All 50 hashes resolved this way; SHA-256 of returned
   bytes matches the walker's reported hash 50/50. The extract tool has a real
   bug ‚Äî the wasm-history walker reads the same archive cleanly so it's not an
   archive issue. Likely a `LedgerEntryChange`-discriminator mismatch or the
   `Created` change comes through under a different metadata variant. **Fix as a
   separate code change once the audit is in.**

2. **wasm-history walker only emits transitions, not initial state.** This caused
   the 283 false negatives (contracts with `ranges: null`) and led to my earlier
   incorrect "doc rot" conclusion on Aquarius. The walker should emit the
   *current* hash for any watched contract that had no transitions in the walked
   range, so a single output reflects the complete picture. Tracked.

3. **Three contracts (Aquarius router, Phoenix multihop anchor, Aquarius admin
   anchor) returned `not_found` from the RPC current-state query.** Their
   ContractInstance entries were TTL-evicted at query time. They're operationally
   live (we ingest through them) ‚Äî TTL evict only the durable storage entry, not
   the contract address. To capture their WASM hashes we'd need either (a) to
   extend the entry's TTL via an invocation and re-query, or (b) to walk the
   archive for their deploy-ledger entry. Outside this audit's scope; document
   as residual.

4. **The 532-contract input list isn't curated in repo.** Future audits would
   have to reconstruct it. Curate `configs/audit/wasm-walk-contracts.yaml` per
   source with provenance.
